# `related_langs/` — language registry + genetic/typological relatedness

Code-keyed reference data answering **"for language X, which other languages are its closest
relatives?"** — used to pick reference languages for per-stem gloss generation today, and
intended as a general, queryable language registry as coverage grows toward thousands of
languages.

> Sibling folder [`regional_langs/`](../regional_langs/README.md) handles the *other* axis:
> regional / script / locale variants **within** one language (e.g. `cmn` → `zh-Hans` /
> `zh-Hant`, `nor` → `nob` / `nno`). Keep the two relations separate — they answer different
> questions and come from different sources.

## Current state (curated bootstrap)

Generated by `bcv-RAG/scripts/build_related_langs.py` from a hand-curated taxonomy of ~50
languages. This is a **bootstrap**, not the final dataset — good enough to drive the gloss
runs now, structured so the scale-up is a data swap, not a redesign.

| file | what |
|---|---|
| `languages.tsv` | registry — one row per ISO 639-3 language (the canonical key) |
| `related.tsv` | relatedness edges — `iso639_3 · rank · related_iso639_3 · basis` |
| `related_languages.json` | the full structure, keyed by `iso639_3` |
| `recommended_refs.tsv` | **derived view** — per target: `ideal_ref_codes` + `available_ref_codes` (availability intersected with generated glosses; convenience only, regenerated) |

### Keys (already future-proof)
- **`iso639_3`** — primary key. The comprehensive 3-letter standard (~7,900 languages).
  639-1 (`iso639_1`, ~180 majors) is a secondary column, blank where none exists.
- **`scripts`** — ISO 15924 codes (`Latn`, `Cyrl`, `Hans`/`Hant`, `Ethi`, `Arab`…).
- **`macrolanguage`** — the ISO 639-3 macrolanguage this individual belongs to (e.g. `cmn`
  → `zho`). The hook for "regional languages under one code" — resolved in `regional_langs/`.
- **`glottocode`** — present as an (empty) column now; filled by the Glottolog ingest below.
- **`stock` / `group` / `branch`** — genetic classification (e.g. Indo-European / Germanic /
  North). `group`+`branch` drive the curated relatedness ordering.

### Design rule: the registry is availability-agnostic
`related.tsv` is **pure linguistics** — it never encodes which languages we happen to have
glosses for. Consumers (the gloss pipeline) intersect with what's on disk at use time. The
`available` column in `languages.tsv` and `available_ref_codes` in the view are *derived
conveniences*, recomputed on every build — never edit them by hand.

## How it's used today

`build_perstem_glosses_llm.py --refs <codes-or-names>` glosses each Hebrew verb stem anchored
on the Hebrew, with **reference languages** as aids; closely-related languages beat English.
Look up the target's row in `recommended_refs.tsv`, pass `available_ref_codes`:

```bash
# nor → dan,deu,nld,eng ;  ara → amh,eng ;  ita → spa,por,fra,eng
python3 scripts/build_perstem_glosses_llm.py Norwegian --refs Danish,German,Dutch,English
```

(The pipeline currently takes display names; mapping codes ↔ names is in `languages.tsv`.
A small code-accepting shim is a planned follow-up.)

## Planned implementation (scale to thousands)

The curated taxonomy doesn't reach thousands — that bulk must be **derived from authoritative
datasets**, keyed by the same `iso639_3`. The plan, in build order:

### 1. Registry from standards
- **ISO 639-3 tables** (SIL, free): the full code list + names + macrolanguage mapping →
  fills `languages.tsv` for all ~7,900 languages.
- **Glottolog** (CC-BY-SA): languoid tree + glottocodes + ISO mappings → fills `glottocode`,
  `stock`/`group`/`branch` genuinely (not hand-typed).

### 2. Relatedness from data (replaces `basis=curated`)
- **Glottolog tree**: shared-node depth between two languages = genetic distance → ranked
  `related.tsv` edges (`basis=genetic-glottolog`).
- **URIEL / lang2vec** (pip, open): precomputed genetic + geographic + syntactic distance
  vectors, purpose-built for choosing transfer-learning "related languages." For a target,
  sort all others by distance, keep top-N (`basis=distance-uriel`). Optionally **ASJP** for
  lexical similarity.
- Curated edges survive as `overrides.tsv` (small, high-confidence, layered on top).

### 3. Served artifact: `languages.db` (SQLite)
At thousands-scale with multiple query patterns, compile the TSV sources into an indexed
SQLite DB (matches the repo's `TSV source → built .db` pattern: spine.db, lxx.db). TSVs stay
the git-diffable source of truth; the DB is the fast lookup.

```sql
-- query by code → ordered relatives  (O(1) indexed)
CREATE TABLE language(iso639_3 TEXT PRIMARY KEY, iso639_1, name, glottocode,
                      stock, group_, branch, scripts, macrolanguage);
CREATE TABLE relatedness(iso639_3, related_iso639_3, rank, distance REAL, basis);
CREATE INDEX rel_by_code ON relatedness(iso639_3, rank);
```

### Query patterns to support
1. **token-language → related languages** (gloss runs): `SELECT related_iso639_3 FROM
   relatedness WHERE iso639_3=? ORDER BY rank`.
2. **by 3-letter code → relatives** (general lookup): same, the registry is code-first.
3. **by code → regional variants**: a join into `regional_langs/` (sibling) — macrolanguage
   members + BCP-47 locale/script variants.

```bash
python3 bcv-RAG/scripts/build_related_langs.py     # rebuilds the four data files
```
Re-run after adding a gloss language (availability columns refresh). The two README.md files
are hand-authored and not touched by the build.
